# Solo Task Creation Integration Plan

## Goal
Add real Solo task creation to FleetTools and make the workflow easy to reuse through a dedicated `solo-task-creation` skill.

## Key Findings
- Solo exposes task creation through the CLI, not HTTP. `solo task create` is implemented in `/Users/johnferguson/Github/Solo/cmd/solo/main.go:149-198`.
- Supported create flags are `--title`, `--type`, `--priority`, `--description`, `--acceptance-criteria`, `--definition-of-done`, `--parent`, `--labels`, `--affected-files`, and `--deps`: `/Users/johnferguson/Github/Solo/cmd/solo/main.go:155-186`.
- Solo task creation returns the normal JSON envelope and actual task payload is nested under `data.task`: `/Users/johnferguson/Github/Solo/internal/solo/tasks.go:73-83`.
- New Solo tasks start as `draft`; making them ready is a separate explicit step: `/Users/johnferguson/Github/Solo/skills/solo/SKILL.md:35-46`.
- FleetTools already has a CLI-shelling `SoloAdapter`, but it lacks `createTask`: `packages/core/src/integrations/solo-adapter.ts:40-114`.
- Fleet Steer currently writes mock handoff files in `packages/cli/src/steer/handoff.ts:6-26`; verify pass calls the mock path in `packages/cli/src/steer/verify.ts:124-135`.

## Recommended Approach
Implement this as both code and a skill:

1. **Code adapter first** so FleetTools can create canonical Solo tasks programmatically and stop relying on mock/file handoff.
2. **Global skill second** so future agents/users can invoke the Solo task creation workflow consistently from OpenCode or Claude Code.

Do not add an HTTP integration. No Solo HTTP task creation endpoint was found.

## Files To Modify

### FleetTools Core
- `packages/core/src/integrations/solo-types.ts`
  - Add `SoloCreateTaskInput` with camelCase fields.
  - Add `SoloCreatedTask` or `SoloTaskCreateData` matching `data.task` shape.
  - Include priority as `number | string` only if the adapter normalizes it; prefer adapter input accepts `priority?: string | number` and serializes to CLI string.
- `packages/core/src/integrations/solo-adapter.ts`
  - Add `createTask(input: SoloCreateTaskInput): Promise<SoloCreatedTask>`.
  - Implement with `solo task create ... --json` through the existing `run<T>()` helper.
  - Convert arrays to comma-separated values for `--labels`, `--affected-files`, and `--deps`.
  - Return `data.task`; throw a clear `SoloCommandError` if `data.task` is missing.
- `packages/core/src/index.ts`
  - Export new Solo create-task types.
- `packages/core/tests/solo-adapter.test.ts` or similar
  - Add tests around CLI arg generation and response parsing using a fake `solo` executable/script.

### Fleet Steer CLI
- `packages/cli/src/steer/types.ts`
  - Add `solo_task_id?: string` and possibly `solo_status?: string` to `TaskStatusFile`.
- `packages/cli/src/steer/handoff.ts`
  - Replace/extend `mockHandoff()` with a real `createSoloTaskHandoff()` that uses `SoloAdapter.createTask()`.
  - Keep mock fallback behind an explicit env/config switch for local tests, e.g. `STEER_SOLO_MOCK=1`.
  - Write a handoff artifact containing Solo task id, status, verify result path, source change slug, and source Steer task id.
  - Preserve failure semantics: Solo creation failure leaves task `done-pending-handoff`.
- `packages/cli/src/steer/verify.ts`
  - On diff-review pass, call the new real handoff function.
  - Store `solo_task_id` on success.
  - Keep current retry behavior for `done-pending-handoff`.
- `packages/cli/src/commands/steer.ts`
  - Ensure `fleet steer handoff --retry` uses the real Solo creation path unless mock mode is explicitly enabled.
- `packages/cli/tests/commands/steer/`
  - Add tests for real handoff path with fake Solo binary.
  - Add tests for failure -> `done-pending-handoff` and retry -> `done`.

### Reusable Skills
Create a global skill for repeatable Solo task creation guidance:

- OpenCode: `/Users/johnferguson/.config/opencode/skill/solo-task-creation/SKILL.md`
- Claude Code: `/Users/johnferguson/.claude/skills/solo-task-creation/SKILL.md`

Skill content should include:
- Trigger phrases: “create Solo task”, “add to Solo”, “turn this plan into Solo tasks”, “handoff verified Steer task”.
- Required fields: title and enough description to make the task actionable.
- Optional fields: type, priority, acceptance criteria, definition of done, labels, affected files, parent, deps.
- Contract: use `solo task create --json`, not HTTP.
- Important behavior: new tasks start as `draft`; `solo task ready <id> --version <version> --json` is a separate explicit action.
- JSON response shape: read created task from `data.task`.
- Verification commands: `solo task show <id> --json` and, if needed, `solo task list --json`.
- Safety rule: do not create in-flight Fleet Steer work in Solo; create tasks only from verified handoff or explicit user request.

## Field Mapping For Fleet Steer Handoff
Use this initial mapping when converting a verified Steer task to a Solo task:

| Fleet Steer Source | Solo Create Field |
|---|---|
| `TaskStatusFile.description` | `title` |
| source change slug + task id + verify summary | `description` |
| `task` | `type` |
| `medium` | `priority` |
| checklist/shell criteria from `verify.md` | `acceptanceCriteria` |
| “Verify result exists and diff review passed” + verify result path | `definitionOfDone` |
| source change slug, `fleet-steer`, `verified` | `labels` |
| declared files from `verify.md` plus changed files if available | `affectedFiles` |
| mapped Solo task ids for completed dependency tasks, if known | `deps` |

MVP should create Solo tasks as `draft`. Do not auto-run `solo task ready` unless the user later asks for that behavior explicitly.

## Implementation Steps
1. Add Solo create-task types and `SoloAdapter.createTask()` with tests in `packages/core`.
2. Add a small mapper in `packages/cli/src/steer/handoff.ts` from Steer task/status/verify files to `SoloCreateTaskInput`.
3. Replace Steer mock handoff default with real `SoloAdapter.createTask()`; keep `STEER_SOLO_MOCK=1` for tests and offline development.
4. Persist `solo_task_id` and handoff artifact on success; preserve `done-pending-handoff` on failure.
5. Update `fleet steer handoff --retry` to use the same real creation path and update task status to `done` only after Solo succeeds.
6. Add focused tests with fake Solo binary and existing mock mode.
7. Create the global `solo-task-creation` skill in both OpenCode and Claude skill locations.
8. Add or update docs explaining that FleetTools creates Solo tasks through the Solo CLI and that created tasks are drafts.

## Risks And Guardrails
- Do not invent an HTTP API. Use the confirmed Solo CLI contract.
- Do not auto-mark tasks ready in this slice; created Solo tasks are draft by design.
- Preserve existing Fleet Steer mock behavior for tests/offline use only.
- Avoid duplicate Solo tasks on retry if a previous success wrote `solo_task_id`; retry should be idempotent when possible.
- Preserve current worktree changes and do not revert unrelated files.
